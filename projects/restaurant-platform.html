Restaurant Ordering Platform

End-to-end information system analysis and design

A web-based restaurant ordering and management system designed to eliminate the operational bottlenecks that occur during peak service hours. The system replaces manual order-taking with a digital platform supporting online ordering, QR-code-based menus, real-time order tracking, and digital payment processing.

The Problem

During peak hours, restaurants face a structural information problem: orders, payments, and preparation status are managed manually, making it difficult to coordinate activities when demand is highest. Customers queue to place orders, staff manage multiple tasks simultaneously, and errors increase under pressure.

This system was designed to address these bottlenecks by creating a structured, real-time information flow between customers, kitchen staff, and management.

How It Works

The system was designed using the full Systems Development Life Cycle (SDLC). The analysis phase identified seven key business events — including order placement, payment processing, payment failure recovery, loyalty point redemption, pre-order scheduling, order status updates, and low-stock alerts — each mapped to a system response using Event Decomposition Technique.

Use case analysis produced UML diagrams for two subsystems: Customer Ordering and Payment, and Staff and Restaurant Operations. These were supported by detailed activity diagrams, system sequence diagrams, and design-level sequence diagrams modelling the complete interactions between six classes: Customer, Restaurant Ordering System, Order, MenuItem, OrderItem, and Staff.

The design phase produced CRC cards assigning class responsibilities, a full design class diagram with methods derived from sequence diagram analysis, and a domain model updated with attributes identified through the design process.

Key Design Decisions

The Restaurant Ordering System was modelled as a GRASP Controller — a design pattern that keeps domain classes focused on their own responsibilities by centralising workflow coordination. This reduces coupling between classes and improves maintainability.

Payment processing was separated from order management as a distinct class, with a further separation between the internal Payment class and the external Payment Gateway. This means a change in payment provider would only affect the boundary between these two classes, leaving the rest of the system unchanged.

A retry loop was built into the payment flow to handle declined transactions gracefully, and atomic transaction management was specified to ensure that a partial system failure cannot leave an order marked unpaid while a payment is recorded as approved.

Risk Analysis

Two critical risk areas were identified: scalability and data integrity. During peak periods, concurrent order placements can create bottlenecks. The modular design — where Order, Payment, and MenuItem operate independently with clearly defined responsibilities — supports future deployment on cloud infrastructure with auto-scaling.

Database constraints including primary keys, foreign keys, and enumerated status values prevent invalid state transitions and protect data consistency.

Cultural Context

The system was designed with reference to Te Ao Māori principles. Manaakitanga — care, respect, and hospitality — informed the core purpose of reducing wait times and improving the customer experience. Whanaungatanga — relationships and belonging — is reflected in the loyalty points feature, which rewards returning customers and fosters an ongoing connection between the restaurant and its community.
