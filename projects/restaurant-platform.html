Restaurant Ordering Platform

End-to-end information system analysis and design

An end-to-end information systems analysis and design project for a web-based restaurant ordering and management platform. The work spans the full Systems Development Life Cycle — from problem identification and requirements analysis through object-oriented design — producing a suite of professional UML artefacts ready to guide software implementation.

The Problem

During peak service hours, restaurants face a structural information failure: orders, payments, and preparation status are managed manually, creating coordination breakdowns when demand is highest. Customers queue, staff manage competing tasks simultaneously, and order errors increase. The system was designed to replace this with a structured, real-time information flow between customers, kitchen staff, and management.

Analysis (Assessment 1)

Seven business events were identified and decomposed using Event Decomposition Technique, each mapped to a system response. Two UML use case diagrams were produced — one for Customer Ordering and Payment, one for Staff and Restaurant Operations. Activity diagrams, system sequence diagrams, a domain model class diagram, and a full entity relationship diagram were developed, establishing the data structure and system boundary.

Design (Assessment 2)

CRC card analysis assigned class responsibilities across five domain classes: Customer, Order, OrderItem, MenuItem, and Staff. Design-level sequence diagrams were produced for both use cases — Place Online Order and Make Payment — using UML 2.0 notation including loop fragments, alt fragments, and break fragments to model retry behaviour and alternative flows.

A unified design class diagram extended the domain model with methods derived directly from sequence diagram analysis. A PaymentGateway class was introduced to establish a clear system boundary with external financial services — meaning a change in payment provider would only require updating the Payment-to-Gateway interaction, leaving the rest of the system unchanged.

Key Design Decisions

The Restaurant Ordering System was modelled as a GRASP Controller — centralising workflow coordination so domain classes remain focused on their own data and behaviour. Payment processing was separated from order management, with the external payment gateway modelled as a distinct class. Atomic transaction management was specified for the payment flow to prevent data inconsistency from partial failures. A retry loop handles declined transactions without breaking the user experience.

Risk Analysis

Concurrent order placement during peak periods was identified as a scalability risk. The modular class design supports future deployment on auto-scaling cloud infrastructure. Database constraints — primary keys, foreign keys, and enumerated status values — prevent invalid state transitions. The isAvailable attribute on MenuItem addresses inventory race conditions under concurrent load.

Cultural Context

The system design drew on Te Ao Māori principles. Manaakitanga informed the core purpose of reducing wait times and improving the customer experience. Whanaungatanga is reflected in the loyalty points feature, which rewards returning customers and builds an ongoing relationship between the restaurant and its community.
